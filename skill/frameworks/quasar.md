# Quasar Framework Setup Rules

[Quasar](https://github.com/blueshift-gg/quasar) is a high-performance, zero-copy, zero-allocation `no_std` Rust framework for Solana programs. It provides Anchor-like ergonomics (`#[program]`, `#[account]`) while achieving extremely low compute unit (CU) overhead, similar to Vanilla/Pinocchio.

When generating a frontend for a Quasar program, you must recognize that while the Rust backend is heavily optimized, the client-side interaction pattern remains remarkably similar to Anchor because Quasar generates compatible IDLs.

## 1. IDL & Client Recognition
- **Identification:** Quasar programs export standard JSON IDLs. If the user specifies the program was built with Quasar, you can largely treat the frontend initialization similar to an Anchor program.
- **Client Libraries:** Unlike pure Vanilla programs that might require manual layout building, Quasar IDLs can typically be consumed by standard TypeScript client generators or the `@coral-xyz/anchor` Typescript SDK.

## 2. Quasar Provider Setup
Even though Quasar is highly optimized on-chain, you can utilize the Anchor SDK to interact with it on the client side, leveraging the generated IDL.

```typescript
import { Program, AnchorProvider } from "@coral-xyz/anchor";
import { useConnection, useWallet } from "@solana/wallet-adapter-react";
import { useMemo } from "react";
// Import the IDL generated by the Quasar build process
import type { MyQuasarProgramType } from "../types/my_quasar_program"; 
import idl from "../idl/my_quasar_program.json";

export function useQuasarProgram() {
  const { connection } = useConnection();
  const wallet = useWallet();

  const program = useMemo(() => {
    if (!wallet || !wallet.publicKey) return null;
    
    const provider = new AnchorProvider(
      connection, 
      wallet as any, 
      { commitment: 'confirmed' }
    );
    
    // Instantiate the program wrapper using the Quasar IDL
    return new Program<MyQuasarProgramType>(idl as MyQuasarProgramType, provider);
  }, [connection, wallet]);

  return program;
}
```

## 3. Data Deserialization Nuances
- Because Quasar relies heavily on zero-copy access (`bytemuck`) on-chain, the data structures defined in the IDL are exactly aligned with memory. 
- **Padding:** Be aware that zero-copy structs may include explicit byte padding to align memory. When consuming the data via React Query, ensure you explicitly map only the relevant fields to the UI and ignore trailing padding bytes.
- **Enums/Unions:** Handle complex nested enums carefully. If the Quasar IDL defines strict C-repr enums, ensure your TypeScript interfaces perfectly map to the integer discriminators.

Once the Program context is established, proceed to [generators/hooks.md](../generators/hooks.md) to build the React Query hooks as you would for any standard Solana program.
